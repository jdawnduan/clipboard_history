Two Loops, One Mission

The clipboard history app runs concurrently — two different things are happening at the same time:

🔄

Monitoring Loop

Every 500ms, checks if the clipboard changed. Like a security guard doing rounds.

👂

Event Loop

Waits for you to press the hotkey. Like a phone that only rings when someone calls.

The Monitoring Loop in Action

Here's the actual code that watches for clipboard changes:

CODE 
async fn monitor_clipboard() {
  let mut history = ClipboardHistory::load()?;
  let mut last_content: Option<String> = None;
  
  loop {
    if let Ok(content) = get_clipboard() {
      let is_new = last_content != Some(&content);
      if is_new && !content.is_empty() {
        history.add(content, MAX_HISTORY_SIZE);
        history.save()?;
      }
    }
    sleep(500).await;
  }
}
PLAIN ENGLISH

Start the clipboard watching function...

Load existing history from disk...

Keep track of what we saw last time (so we know if it's NEW)

Start an infinite loop — this never stops!

Try to read the clipboard. If it works...

Is this different from what we saw last time?

If it's new AND not empty, save it!

Save the updated history to disk...

Wait 500ms (half a second), then check again

This closing brace matches the 'loop' opening brace

Why Polling? Why Not Events?

You might wonder: why does the app "check" every 500ms instead of being notified immediately when something changes?

The answer is the clipboard API doesn't support event-based programming. There's no "tell me when clipboard changes" feature.

So the app has to check periodically, like:

ℹ️
Good to know

500ms was chosen as a balance: fast enough to feel instant when you paste, but not so fast that it wastes battery or CPU.
