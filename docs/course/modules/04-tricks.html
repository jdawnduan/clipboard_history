The Clever Tricks

Engineering decisions that make the app reliable

Four Smart Decisions

Building a clipboard manager sounds simple, but there are tricky edge cases. Here are the clever solutions:

🚫

Duplicate Prevention

If you copy the same text twice, only the newest copy is kept — like a "most recent" filter.

📏

Size Limits

Entries over 128KB are skipped — prevents huge images or files from bloating storage.

🔒

Single Instance

Only one clipboard history daemon runs at a time — using a lock file on disk.

⏭️

Paste Skip Flag

When pasting from history, the monitor ignores that clipboard change — no infinite loops!

The Skip Flag — Avoiding Infinite Loops

Here's one of the trickiest problems: when you select from history, the app writes to the clipboard. But the monitor is watching the clipboard! It would see the "paste" as a new copy and add it back to history...

That's where the skip flag comes in:

CODE 
// When starting paste:
skip_next_monitor.store(true, Ordering::SeqCst);

// Inside monitor loop:
if skip_next.swap(false, Ordering::SeqCst) {
  last_content = Some(content);
  continue;  // Skip adding to history!
}
PLAIN ENGLISH

When user clicks to paste from history...

Set the skip flag to TRUE — "don't count the next clipboard change!"

Empty line for readability

Meanwhile, in the monitoring loop (every 500ms)...

Check the flag and RESET it to false at the same time...

If it was true, update what we last saw but...

Skip adding this to history! Go back to watching.

Single Instance — No Duplicate Daemons

What if you accidentally start clipboard history twice? You'd have two daemons fighting over the clipboard. The solution: a lock file.

The History Entry Structure

Each entry in your clipboard history is a simple data structure:

CODE 
pub struct ClipboardEntry {
  pub content: String,       // The actual text
  pub timestamp: DateTime<Utc>, // When it was copied
}
