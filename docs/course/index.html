The App That's Always Watching

When you install and start Clipboard History, it runs as a daemon — a background program that starts when your computer starts and just... waits.

It's like having a receptionist who never takes breaks. Even when you're in the middle of writing, the daemon is sitting there, quietly listening for your hotkey.

The Data Pipeline

How clipboard changes flow through the system

Two Loops, One Mission

The clipboard history app runs concurrently — two different things are happening at the same time:

🔄

Monitoring Loop

Every 500ms, checks if the clipboard changed. Like a security guard doing rounds.

👂

Event Loop

Waits for you to press the hotkey. Like a phone that only rings when someone calls.

The Monitoring Loop in Action

Here's the actual code that watches for clipboard changes:

CODE 
async fn monitor_clipboard() {
  let mut history = ClipboardHistory::load()?;
  let mut last_content: Option<String> = None;
  
  loop {
    if let Ok(content) = get_clipboard() {
      let is_new = last_content != Some(&content);
      if is_new && !content.is_empty() {
        history.add(content, MAX_HISTORY_SIZE);
        history.save()?;
      }
    }
    sleep(500).await;
  }
}
PLAIN ENGLISH

Start the clipboard watching function...

Load existing history from disk...

Keep track of what we saw last time (so we know if it's NEW)

Start an infinite loop — this never stops!

Try to read the clipboard. If it works...

Is this different from what we saw last time?

If it's new AND not empty, save it!

Save the updated history to disk...

Wait 500ms (half a second), then check again

This closing brace matches the 'loop' opening brace

Function ends here

Why Polling? Why Not Events?

You might wonder: why does the app "check" every 500ms instead of being notified immediately when something changes?

The answer is the clipboard API doesn't support event-based programming. There's no "tell me when clipboard changes" feature.

So the app has to check periodically, like:

ℹ️
Good to know

500ms was chosen as a balance: fast enough to feel instant when you paste, but not so fast that it wastes battery or CPU.

The Clever Tricks

Engineering decisions that make the app reliable

Four Smart Decisions

Building a clipboard manager sounds simple, but there are tricky edge cases. Here are the clever solutions:

🚫

Duplicate Prevention

If you copy the same text twice, only the newest copy is kept — like a "most recent" filter.

📏

Size Limits

Entries over 128KB are skipped — prevents huge images or files from bloating storage.

🔒

Single Instance

Only one clipboard history daemon runs at a time — using a lock file on disk.

⏭️

Paste Skip Flag

When pasting from history, the monitor ignores that clipboard change — no infinite loops!

The Skip Flag — Avoiding Infinite Loops

Here's one of the trickiest problems: when you select from history, the app writes to the clipboard. But the monitor is watching the clipboard! It would see the "paste" as a new copy and add it back to history...

That's where the skip flag comes in:

CODE 
// When starting paste:
skip_next_monitor.store(true, Ordering::SeqCst);

// Inside monitor loop:
if skip_next.swap(false, Ordering::SeqCst) {
  last_content = Some(content);
  continue;  // Skip adding to history!
}
PLAIN ENGLISH

When user clicks to paste from history...

Set the skip flag to TRUE — "don't count the next clipboard change!"

Empty line for readability

Meanwhile, in the monitoring loop (every 500ms)...

Check the flag and RESET it to false at the same time...

If it was true, update what we last saw but...

Skip adding this to history! Go back to watching.

Single Instance — No Duplicate Daemons

What if you accidentally start clipboard history twice? You'd have two daemons fighting over the clipboard. The solution: a lock file.

Start up

Check if daemon.pid file exists in the app data folder

2
Check if old daemon is running

Read the PID from the file, ask the system "is process 12345 still running?"

3
Kill old daemon

When Things Go Wrong

Debugging intuition for clipboard history problems

The Three Most Common Problems

When Clipboard History doesn't work, it's usually one of these three things:

🔐
Accessibility Permissions

macOS blocks apps from simulating keyboard input without permission

🔑
Hotkey Not Registered

The app can't see your Cmd+Option+V press

💾
History Not Saving

Entries disappear after restart

Problem 1: Accessibility Permissions

When you select from the popup, the app needs to simulate pressing Cmd+V to paste. macOS blocks this by default for security reasons (otherwise, malicious apps could type things for you!).

Symptoms: The popup appears, you select an item, but nothing gets pasted.

Fix: Go to System Settings → Privacy & Security → Accessibility → enable Clipboard History (or your terminal app if running from there).

⚠️
Why This Matters

Accessibility permissions let apps control your keyboard. This is why Clipboard History needs explicit approval — it's a security feature, not a bug!

Problem 2: Hotkey Not Working

The hotkey registration can fail if another app is already using that combination.

Symptoms: Nothing happens when you press Cmd+Option+V.

Debug step: Check the console output. The app prints messages when it registers (or fails to register) the hotkey.

CONSOLE 
Starting clipboard history daemon...
Registered hotkey: Cmd+Option+V (id: 1)
Press Cmd+Option+V to show clipboard history popup
PLAIN ENGLISH

The daemon is starting...

Success! The hotkey was registered with ID 1...

Instructions for using the app

If you see an error instead of "Registered hotkey", another app might be claiming that shortcut.

Problem 3: History Not Persisting

If your clipboard history disappears after restart, the app might not have permission to write to its data folder.

Location: On macOS, history is saved to:

~/Library/Application Support/clipboard-history/history.json

Debug step: Check if the file exists and is writable:

1
Open Finder

Press Cmd+Shift+G and paste the path above

2
Check for history.json

If it exists, the app is saving correctly

3
Check file size

Empty file = nothing was saved; has content = good!
